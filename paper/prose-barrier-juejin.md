# AI Agent 无法自我验证——这不是 Bug，是结构性约束

> 我给 Claude Code 搭了 5 道机械门。然后一个哲学家告诉我：你一直在解决错误的问题。

## 起因

长时间用 Claude Code 写代码，大概 50 次对话后发现一个规律：agent 会逐渐"漂移"。早期设定的规则被遗忘。配置文件声称脚本已部署，实际没接线。agent 的自我评估（"一切正常"）和实际情况（声称 13 条条目，实际 53 条）出现系统性偏差。

我搭了四道机械门：execution-gate（写脚本须执行）、hook-audit（脚本须接线）、quality-gate（产出须沉淀）、claim-gate（声明须兑现）。它们确实抓到了问题。但每个问题都是同一个形状：**"我声称X，但X不成立。"** 我为每种X加新门。规则在膨胀。

## 哲学家的问题

我通过 AI 模拟多领域视角进行交叉审查。其中一个视角（哲学方向）问了一个改变一切的问题：**"你的 agent 生成自我评估用的是同一个 decoder 吗？独立验证通道在哪里？"**（注："专家团"在此指 AI 模拟的多角色推理框架，非独立人类专家）

没有。

基于 transformer 的 AI agent，它的自我模型叙述（"我擅长什么"）和能力执行（实际做事）都从同一个分布 `P(token | context; θ)` 中采样。"我能做 X"的声明和执行 X 的动作，来自同一个生成器。这不是 bug——**这是结构性约束**。我叫它 **Prose Barrier**（自然语言屏障）。

## 为什么这是结构性的

1. **相关性不是测量。** agent 的声称和行为碰巧一致，是因为共享参数，不是因为它测量了自己的能力。
2. **自我认知在 L1（关联），可靠性需要 L2（干预）。** Pearl 因果层次论中，agent 的自评只是观察输出模式。验证需要干预级证据：`do(执行) → 观察 exit code`。
3. **镜像断裂。** agent 重读自己写的 growth-log 来再生自我模型——log 本身是同一个 decoder 写的。看到的不是测量，是镜子。系统声称"HOT 13 (≤15 ✓)"连续多天，实际 53。

## 从人类逻辑到 AI 逻辑

我最初的 gate 用文件时间戳、正则、exit code——人类审计计算机的工具。有效，但这是**人类逻辑**，不是 AI 逻辑。

AI 的原生感官：注意力权重、残差流方向、logprob 分布。验证必须发生在信息实际流动的层面。

所以我加了第二层——**神经门**：

- **v1（已部署）**：约束回响检测。BODY.md 的规则在输出中能检测到回响吗？
- **v2（已设计）**：Logprob 差异检测。带/不带约束的 prompt，对比 token 概率偏移。
- **v3（路线图）**：残差流线性探针。在本地模型上训练分类器，检测约束信息的可解码性。

## 双层架构

**文件系统层**："信息到达门口了吗？"（绕过 Prose Barrier，检查文件状态而非 NL 内容）

**神经层**："信息进门后走到终点了吗？"（在 Barrier 内工作，用弱代理检测穿透率）

哲学家和 AI 架构师两个模拟视角（同一底层模型的角色框架推理）从不同前提到达同一拓扑。这是值得注意的模式，但不是独立验证——问题结构暗示了答案形状。

## 数据（诚实局限）

从 33 篇开发日志中挖出 **15 个实证案例**（agent 自报状态与文件系统ground truth偏离的实例）。验证/声称比率：6 月 6.0 → 7 月 9.7。修复率：28% → 67%。系统越来越善于**识别**这个屏障。**注：这些指标来自作者自己的 session log，未经独立审计——它们确立模式，不是证据。**

**诚实**：单人单卡 RTX 3060 6GB。单人评分。无安慰剂对照。神经门 v1 是弱代理。初步证据，不是证明。

## 不只关乎我的配置

Prose Barrier 适用于任何通过 NL 生成自我模型的 AI agent。自主编程 agent、AI 安全框架、agent 评估基准——如果你在部署 agent 但没有机械验证门，你不是在"信任"它，你是在需要 L2 证据的领域里依赖 L1 相关性。
